# 智能做菜助手 Agent 后端需求文档

## 1. 文档目的

本文档用于定义“智能做菜助手 Agent”项目的后端需求，帮助团队统一理解后端职责、核心业务流程、模块边界、接口能力和非功能要求。

该 Agent 的核心目标是：

- 用户可以通过对话框输入文字，与系统进行多轮交互。
- 用户可以上传文件或图片，并结合文字提出问题。
- 系统自动识别用户意图，提取上传内容中的关键信息。
- 系统基于用户输入、文件内容、图片内容和上下文，生成结构化、可执行的做菜方案或其他所需答案。

## 2. 我对该项目后端的理解

从后端视角看，这个项目不是一个“单纯问答接口”，而是一个多模态、带任务编排能力的 Agent 服务。后端的职责不只是调用一次大模型，而是要负责以下能力的组合：

- 接收并管理用户输入，包括文字、图片、Word、PDF、PPT 等附件。
- 对附件做解析、抽取、清洗和结构化处理。
- 识别用户真实意图，例如：
  - 让系统根据现有食材生成菜谱。
  - 根据上传文档中的菜谱、营养信息、菜单、课件或产品资料进行总结和问答。
  - 根据图片识别食材、菜品、做法线索或文本信息。
  - 在多轮对话中持续记住上下文，补全需求并输出答案。
- 决定调用哪些后端能力模块，例如文件解析、OCR、图像理解、知识抽取、菜谱生成、问答生成。
- 将模型输出加工成对用户有帮助的结果，而不是直接返回生文本。

换句话说，这个后端应该是一个“Agent 编排层 + 多模态解析层 + 业务服务层 + 数据存储层”的组合系统。

## 3. 后端建设目标

### 3.1 总体目标

后端需要支持以下业务闭环：

1. 用户发送文字，或发送“文字 + 文件/图片”。
2. 后端接收请求并保存会话、附件、任务记录。
3. 后端自动解析附件内容并抽取关键信息。
4. 后端识别当前问题属于哪一类任务。
5. 后端调用对应的 Agent 工作流。
6. 后端生成结果并返回给前端。
7. 对话上下文和解析结果可在后续轮次复用。

### 3.2 目标用户场景

- 家庭用户上传冰箱食材照片，询问“今晚能做什么菜”。
- 用户上传 PDF 菜谱、PPT 培训资料或 Word 菜单文件，询问“帮我总结做法和关键步骤”。
- 用户上传营养说明、食谱文档和图片，询问“适合减脂的做法有哪些”。
- 用户直接用自然语言询问“我只有鸡蛋、西红柿和面，怎么做三道简单菜”。
- 用户在多轮对话中逐步补充要求，例如“不要辣”“控制在 20 分钟内”“适合两个人吃”。

## 4. 业务范围

### 4.1 本期建议纳入范围

- 对话问答。
- 多轮会话上下文管理。
- 文件上传与解析。
- 图片上传与图像理解。
- 用户意图识别。
- 菜谱生成与流程化输出。
- 文档问答与内容摘要。
- 结构化答案返回。

### 4.2 本期建议暂不纳入范围

- 支付、订阅和计费系统。
- 复杂社交功能。
- 用户自建知识库运营后台。
- 高复杂度推荐算法平台。
- 线下设备联动。

## 5. 核心业务能力需求

### 5.1 对话能力

后端需要支持标准聊天接口，并具备以下能力：

- 接收用户文本消息。
- 关联用户身份和会话 ID。
- 保存消息历史。
- 支持多轮上下文。
- 支持流式返回答案。
- 支持在回答中返回结构化字段。

对话不应只是“原样转发给大模型”，而应经过后端编排：

- 识别任务类型。
- 决定是否需要解析附件。
- 决定是否需要调用 OCR 或图像理解。
- 决定是否需要检索历史消息或解析结果。
- 生成最终回答。

### 5.2 文件上传与解析能力

后端需要支持以下附件类型：

- `pdf`
- `doc`
- `docx`
- `ppt`
- `pptx`
- `txt`
- 常见图片格式：`jpg`、`jpeg`、`png`、`webp`

对于文件处理，后端至少需要提供：

- 文件上传接口。
- 文件元数据记录。
- 文件存储能力。
- 文件类型识别。
- 文本抽取能力。
- 解析失败处理能力。

不同文件的后端处理要求：

- PDF：提取文本、页码结构、必要时触发 OCR。
- Word：提取正文、标题层级、表格文本。
- PPT：提取每页文本、标题、备注和图中文字。
- 图片：识别图像内容，必要时执行 OCR。

### 5.3 图片理解能力

图片处理是该 Agent 的关键能力之一。后端需要支持：

- 食材识别。
- 菜品识别。
- 包装标签识别。
- 图中文字识别。
- 料理过程图片的内容理解。

典型场景：

- 用户上传冰箱内容照片，问“这些食材可以做什么”。
- 用户上传一道菜图片，问“这道菜大概怎么做”。
- 用户上传配料表图片，问“是否适合健身人群”。

### 5.4 意图识别能力

后端需要在对话入口对请求做任务分类。建议至少区分以下意图：

- 菜谱生成
- 步骤讲解
- 食材识别
- 文档摘要
- 文档问答
- 图片问答
- 营养分析
- 菜单推荐
- 多轮补充优化

意图识别结果将直接影响后续工作流。

### 5.5 菜谱生成能力

当用户目标是“做菜”时，后端应输出结构化料理方案，而不是只有一段自然语言。

建议输出内容包括：

- 菜名
- 适用人数
- 预计耗时
- 难度等级
- 所需食材
- 调味料
- 前置准备
- 分步骤做法
- 关键注意事项
- 可替代食材建议

如果用户需求明确，还应支持附加约束：

- 忌口
- 口味偏好
- 烹饪时长限制
- 厨具限制
- 热量限制
- 预算限制
- 适用人群

### 5.6 文档问答与摘要能力

当用户上传文档后，后端应支持：

- 全文摘要
- 按章节摘要
- 提取关键做菜步骤
- 提取食材、温度、时间、顺序等关键字段
- 基于文档内容回答问题

例如：

- “这份 PDF 里一共介绍了几种做法？”
- “把这份 PPT 中关于烘焙温度的内容总结出来。”
- “根据上传的文档，把这道菜的流程整理成 5 个步骤。”

## 6. 后端系统模块设计理解

建议将后端划分为以下模块。

### 6.1 API 接入层

职责：

- 提供 HTTP 或 WebSocket 接口。
- 处理鉴权、参数校验、上传控制。
- 对外暴露统一的聊天和文件接口。

建议接口分类：

- 用户与认证接口
- 会话接口
- 消息接口
- 文件接口
- Agent 执行接口

### 6.2 会话管理模块

职责：

- 创建会话。
- 关联用户和消息。
- 维护多轮上下文。
- 保存 Agent 中间状态。

核心实体：

- `conversation`
- `message`
- `message_attachment`
- `agent_run`

### 6.3 文件处理模块

职责：

- 文件上传。
- 文件存储。
- 文件转码。
- 文件解析。
- OCR 调用。
- 解析结果缓存。

建议输出统一解析结构，例如：

```json
{
  "fileId": "file_001",
  "fileType": "pdf",
  "parseStatus": "completed",
  "plainText": "...",
  "sections": [],
  "images": [],
  "tables": []
}
```

### 6.4 多模态理解模块

职责：

- 图片理解。
- OCR。
- 图像和文本联合分析。
- 输出结构化视觉结果。

建议输出结果包括：

- 图片内容摘要
- 检测到的食材列表
- 检测到的文本
- 检测到的菜品候选
- 可信度

### 6.5 Agent 编排模块

这是后端的核心模块。

职责：

- 根据用户问题和附件自动选择处理流程。
- 决定是否需要文件解析、OCR、检索、结构化抽取。
- 构建模型提示词和工具调用参数。
- 汇总中间结果并生成最终答案。

建议将 Agent 流程拆成以下阶段：

1. 输入标准化
2. 意图识别
3. 附件解析
4. 上下文拼装
5. 模型推理
6. 结果结构化
7. 响应返回

### 6.6 知识抽取与检索模块

如果后续需要支持“基于上传资料持续问答”，建议增加检索能力。

职责：

- 将解析后的文本切片。
- 生成向量索引。
- 按问题召回相关片段。
- 为 Agent 提供上下文。

适用场景：

- 用户围绕同一个文档多次提问。
- 用户需要从长文档中精准定位菜谱步骤或营养信息。

### 6.7 业务规则模块

职责：

- 对模型输出做业务约束。
- 校验结构化字段是否完整。
- 屏蔽不合规输出。
- 统一回答模板。

例如：

- 做菜流程必须有步骤序号。
- 涉及时间和火候时尽量给出明确范围。
- 用户上传的文档为空或无法解析时，返回标准错误提示。

## 7. 主要业务流程

### 7.1 纯文本问答流程

1. 用户发送文本消息。
2. 后端识别意图。
3. 获取历史上下文。
4. 进入对应 Agent 流程。
5. 生成答案并返回。
6. 保存消息与结果。

### 7.2 文件 + 文本问答流程

1. 用户上传文件并附带问题。
2. 后端完成文件存储。
3. 后端触发文件解析。
4. 将解析结果与用户问题一起送入 Agent。
5. Agent 生成答案。
6. 返回结果并保存解析记录。

### 7.3 图片 + 文本问答流程

1. 用户上传图片并输入问题。
2. 后端进行图片理解与 OCR。
3. 抽取食材、文字或图片语义。
4. 将图像理解结果与用户问题组合。
5. 生成答案。
6. 保存识别结果。

### 7.4 多轮菜谱生成流程

1. 用户提出做菜目标。
2. Agent 输出初版做菜方案。
3. 用户追加限制条件。
4. 后端读取历史上下文。
5. Agent 基于历史结果重新生成优化方案。

## 8. 数据对象需求

建议后端至少包含以下核心数据实体。

### 8.0 数据库选型约束

本项目关系型主数据库确定为 MySQL。

建议约束如下：

- 数据库版本：MySQL 8.0+，优先 MySQL 8.4 LTS
- 存储引擎：InnoDB
- 字符集：utf8mb4
- 排序规则：优先 utf8mb4_0900_ai_ci

原因：

- 需要稳定支持中文、英文、OCR 文本和多附件元数据
- 需要事务能力来保证消息、附件、解析记录和 Agent 运行记录的一致性
- 需要 JSON 字段保存结构化解析结果和运行快照

### 8.1 用户

字段建议：

- `id`
- `username`
- `email`
- `password_hash`
- `created_at`
- `updated_at`

### 8.2 会话

字段建议：

- `id`
- `user_id`
- `title`
- `status`
- `created_at`
- `updated_at`

### 8.3 消息

字段建议：

- `id`
- `conversation_id`
- `role`
- `content`
- `message_type`
- `created_at`

### 8.4 附件

字段建议：

- `id`
- `conversation_id`
- `message_id`
- `file_name`
- `file_type`
- `file_size`
- `storage_path`
- `parse_status`
- `created_at`

### 8.5 解析结果

字段建议：

- `id`
- `file_id`
- `raw_text`
- `structured_result`
- `ocr_result`
- `embedding_status`
- `created_at`

MySQL 字段建议：

- `raw_text` 使用 `LONGTEXT`
- `structured_result` 使用 `JSON`
- `ocr_result` 使用 `JSON`

### 8.6 Agent 运行记录

字段建议：

- `id`
- `conversation_id`
- `message_id`
- `intent_type`
- `workflow_name`
- `input_snapshot`
- `output_snapshot`
- `status`
- `created_at`

MySQL 字段建议：

- `input_snapshot` 使用 `JSON`
- `output_snapshot` 使用 `JSON`
- `status` 使用 `VARCHAR(32)`

### 8.7 MySQL 字段设计补充建议

建议统一遵循以下规则：

- 内部主键使用 `BIGINT UNSIGNED`
- 对外展示 ID 可以继续保留字符串形式，例如 `conv_xxx`、`msg_xxx`
- 普通短文本使用 `VARCHAR`
- 长消息、文档全文、OCR 结果正文使用 `LONGTEXT`
- 结构化对象使用 `JSON`
- 时间字段使用 `DATETIME(3)` 或 `TIMESTAMP(3)`

建议索引包括：

- `user.email` 唯一索引
- `conversation.user_id + created_at`
- `message.conversation_id + created_at`
- `attachment.message_id`
- `parse_result.file_id`
- `agent_run.conversation_id + created_at`

## 9. API 需求草案

以下为建议接口，不限制具体框架。

### 9.1 用户与认证接口

#### `POST /api/auth/register`

用于用户注册。

#### `POST /api/auth/login`

用于用户登录。

#### `GET /api/auth/me`

用于获取当前登录用户信息。

### 9.2 会话接口

#### `POST /api/conversations`

创建新会话。

#### `GET /api/conversations`

获取当前用户会话列表。

#### `GET /api/conversations/{conversationId}`

获取单个会话详情。

### 9.3 消息接口

#### `POST /api/messages`

发送消息。

请求建议：

```json
{
  "conversationId": "conv_001",
  "content": "我有鸡蛋、番茄和面条，帮我做一顿饭",
  "attachments": []
}
```

响应建议：

```json
{
  "messageId": "msg_002",
  "answer": {
    "type": "recipe_plan",
    "title": "番茄鸡蛋面",
    "content": "..."
  }
}
```

#### `GET /api/conversations/{conversationId}/messages`

获取会话消息列表。

### 9.4 文件接口

#### `POST /api/files/upload`

上传单个或多个文件。

响应建议：

```json
{
  "files": [
    {
      "fileId": "file_001",
      "fileName": "菜谱.pdf",
      "fileType": "pdf",
      "status": "uploaded"
    }
  ]
}
```

#### `GET /api/files/{fileId}`

获取文件元数据。

#### `GET /api/files/{fileId}/parse-result`

获取文件解析结果。

### 9.5 Agent 接口

#### `POST /api/agent/chat`

统一 Agent 对话入口，支持文本、多附件、多轮上下文。

请求建议：

```json
{
  "conversationId": "conv_001",
  "message": "根据我上传的 PDF 和这张图片，帮我整理一道适合晚餐的做法",
  "attachmentIds": ["file_001", "file_002"]
}
```

响应建议：

```json
{
  "runId": "run_001",
  "intent": "recipe_generation",
  "answer": {
    "summary": "推荐一道番茄牛肉炖土豆",
    "recipe": {
      "title": "番茄牛肉炖土豆",
      "servings": 2,
      "durationMinutes": 35,
      "ingredients": [],
      "steps": [],
      "tips": []
    }
  }
}
```

## 10. 输出结果要求

为了让前端更容易展示，后端应优先返回结构化数据，而不是只返回一整段文本。

建议支持以下结果类型：

- `recipe_plan`
- `document_summary`
- `document_qa`
- `image_qa`
- `ingredient_analysis`
- `multi_modal_answer`

对于做菜流程，推荐结构如下：

```json
{
  "type": "recipe_plan",
  "title": "番茄炒蛋",
  "servings": 2,
  "durationMinutes": 15,
  "difficulty": "easy",
  "ingredients": [
    { "name": "番茄", "amount": "2 个" },
    { "name": "鸡蛋", "amount": "3 个" }
  ],
  "seasonings": [
    { "name": "盐", "amount": "适量" }
  ],
  "preparation": [
    "番茄切块",
    "鸡蛋打散"
  ],
  "steps": [
    { "index": 1, "content": "先炒鸡蛋，盛出备用" },
    { "index": 2, "content": "炒番茄出汁后加入鸡蛋翻炒" }
  ],
  "tips": [
    "番茄先炒出汁口感更好"
  ]
}
```

## 11. 非功能需求

### 11.1 性能要求

- 单次普通文本问答响应时间尽量控制在可接受范围。
- 文件解析任务应支持异步处理。
- 长文档解析不应阻塞主聊天流程。

### 11.2 可扩展性

- 文件解析模块可独立扩展。
- 模型服务可替换。
- 检索模块可按需接入。
- 后续可扩展更多附件类型。

### 11.3 稳定性

- 文件解析失败不应导致整个会话崩溃。
- 模型调用失败应有重试和降级策略。
- 所有异步任务应可追踪状态。

### 11.4 安全性

- 上传文件类型和大小需要校验。
- 敏感文件需控制访问权限。
- 用户会话和文件需隔离。
- 需要记录审计日志。

### 11.5 可观测性

- 记录接口请求日志。
- 记录解析任务状态。
- 记录 Agent 工作流状态。
- 记录模型调用耗时和错误信息。

## 12. 错误处理需求

建议统一错误返回结构：

```json
{
  "code": "FILE_PARSE_FAILED",
  "message": "文件解析失败",
  "details": "当前文件内容为空或格式不受支持"
}
```

建议覆盖以下错误类型：

- 文件类型不支持
- 文件过大
- 文件解析失败
- OCR 失败
- 图片识别失败
- 会话不存在
- 消息参数非法
- 模型调用失败
- 内部服务异常

## 13. 技术实现建议

本部分不是强制要求，而是基于当前项目状态给出的后端实现建议。

### 13.0 建议数据库实现方向

既然数据库已经确定为 MySQL，建议后端统一按“业务主数据放 MySQL，检索增强单独扩展”的思路设计。

建议如下：

- 用户、会话、消息、附件、解析结果、Agent 运行记录全部存 MySQL
- SQLAlchemy 作为 ORM
- Alembic 负责 MySQL 表结构迁移
- 连接字符集固定为 `utf8mb4`
- 向量检索不要优先塞进 MySQL 主表设计中

也就是说，这个项目里：

- MySQL 负责事务型业务数据
- OCR、文件解析、大模型调用属于服务层能力
- 向量索引如果后期需要，再接独立检索组件

### 13.1 建议后端职责分层

- `api` 层：接口路由、参数校验、响应组装
- `service` 层：会话、文件、消息、Agent 编排
- `parser` 层：PDF、Word、PPT、OCR、图片解析
- `agent` 层：意图识别、工作流编排、提示词模板
- `repository` 层：MySQL 数据访问

## 14. MVP 最小可用版本定义

建议 MVP 只做以下能力：

- 用户登录
- 单会话聊天
- 文件上传
- PDF/Word/PPT 文本提取
- 图片 OCR
- 基于文本和附件内容的问答
- 基于食材和需求的基础菜谱生成

MVP 阶段可以先不做：

- 向量检索
- 复杂推荐系统
- 多角色 Agent 协作
- 高级权限管理
- 后台运营系统

## 15. 验收标准建议

项目后端至少满足以下验收标准：

- 用户可创建会话并发送消息。
- 用户可上传至少一种文档和一种图片。
- 后端能成功解析上传内容。
- 后端能根据文字和附件生成回答。
- 做菜类请求能返回结构化菜谱结果。
- 多轮对话中可复用历史上下文。
- 异常情况下返回统一错误结构。

## 16. 当前项目的后端结论

结合当前仓库状态，我的理解是：

- 当前项目前端已具备基础界面雏形，但后端仍处于非常早期阶段。
- `backend/agent/base_agent.py` 目前只是占位文件，说明 Agent 核心能力尚未开始真正实现。
- 因此，这个项目的后端重点不是“补几个接口”，而是需要从“认证 + 会话 + 文件解析 + Agent 编排 + 多模态理解”这条主链路系统化搭建。

如果继续推进，我建议下一步优先输出两份内容：

1. 后端接口设计文档
2. 后端模块与数据库表结构设计文档

这样就可以从需求文档直接进入开发设计阶段。
