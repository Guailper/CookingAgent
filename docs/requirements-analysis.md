# CookingAgent 生产级需求分析文档

更新日期：2026-05-24

## 1. 文档定位

本文档定义 CookingAgent 从“可本地运行的 AI 做菜助手”升级为“可实际部署、可长期维护、面向生产环境运行的应用”所需满足的产品、后端、前端、安全、性能、可观测和运维需求。

本文档回答三个问题：

- 产品要为用户提供哪些稳定能力。
- 生产环境上线前必须补齐哪些工程能力。
- 如何从功能、效率、安全、用户体验和可运维性角度验收。

架构分层、模块边界和数据流见 [architecture-design.md](architecture-design.md)。

## 2. 产品目标

CookingAgent 是一个面向中文做菜场景的 AI Agent 应用。系统以聊天工作台为入口，结合本地菜谱知识库、用户长期记忆、附件解析、语音转写、联网搜索和天气查询，为用户提供菜谱问答、烹饪建议、菜单规划、食材处理和上下文连续对话能力。

生产级目标：

- **稳定可用**：在真实用户访问、模型波动、外部 API 失败和附件解析耗时的情况下，仍能提供可理解、可恢复的服务体验。
- **响应高效**：普通接口低延迟，Agent 首 token 尽快返回，长耗时任务异步化，缓存和索引服务于真实流量。
- **安全默认**：认证、授权、上传、CORS、限流、日志、密钥、依赖和部署配置默认按生产安全要求设计。
- **体验完整**：用户能清晰感知登录、上传、解析、检索、生成、失败重试和历史会话状态。
- **可运维**：支持容器化部署、数据库迁移、健康检查、日志、指标、告警、备份、回滚和灰度发布。
- **可迭代**：需求、架构、测试和评估闭环清晰，后续可以扩展多知识库、组织空间、任务队列和更强的 Agent 工具。

## 3. 用户与场景

| 用户角色 | 目标 | 典型场景 |
| --- | --- | --- |
| 普通做饭用户 | 快速获得可靠的中文烹饪建议 | 问某道菜怎么做、询问替代食材、根据厨具限制调整步骤 |
| 有长期偏好的用户 | 让系统记住忌口、口味和健康目标 | “我不吃香菜”“以后少放糖”“我家只有空气炸锅” |
| 上传资料的用户 | 让系统理解附件内容并参与回答 | 上传菜谱、菜单、食材清单、图片或文档后继续提问 |
| 移动端/弱网用户 | 在不稳定网络下仍能持续使用 | 流式回答可恢复，上传有进度，失败能重试 |
| 运营与维护者 | 管理知识库、排查运行问题、控制成本 | 重建索引、查看 agent_run 快照、观察 token 成本和错误率 |
| 系统管理员 | 安全部署和持续运维 | 配置密钥、数据库迁移、备份恢复、监控告警、发布回滚 |

## 4. 版本范围

### 4.1 生产 MVP 必须包含

- 认证与账号体系：注册、登录、验证码、当前用户、密码修改、账号状态控制。
- 会话与消息：会话创建、列表、详情、消息发送、消息历史、消息元数据。
- Agent 流式问答：SSE 流式输出、模型 fallback、本地兜底、运行记录。
- RAG 问答：默认菜谱知识库、查询改写、Milvus 检索、rerank、引用快照。
- 长期记忆：偏好/忌口/厨具/健康目标抽取、去重、更新、上下文注入。
- 附件上传与解析：文件上传、未绑定删除、解析状态、文档入库。
- 语音转写：音频上传、大小/时长校验、远程或本地转写。
- 前端聊天工作台：登录、会话、消息流式显示、附件、语音、错误提示、加载状态。
- 安全基线：严格认证授权、限流、CORS allowlist、Host 校验、安全响应头、上传防护、密钥管理。
- 运维基线：Docker 部署、环境配置模板、数据库迁移、健康检查、日志、指标、告警、备份。
- 测试基线：后端单元测试、关键接口集成测试、前端构建、基础端到端冒烟测试。

### 4.2 生产增强项

- 组织空间、多用户协作、角色权限。
- 用户自定义知识库和知识库分享。
- 对附件解析与文档入库引入后台任务队列。
- 对长期记忆引入向量检索。
- 对 Agent 输出增加结构化菜谱卡片、购物清单和营养估算。
- 多模型路由、成本预算、用量报表。
- 灰度发布、A/B 测试和提示词版本管理。
- 对象存储和 CDN 加速。
- 多区域部署和灾备切换。

### 4.3 当前代码已具备的基础

当前仓库已经具备 FastAPI API、SQLAlchemy ORM、LangChain Agent、RAG、长期记忆、附件解析、语音转写、Redis 可选缓存、RAGAS 评估脚本和 React 聊天工作台基础。生产化工作应在此基础上补齐迁移、部署、安全、后台任务、监控和体验细节。

## 5. 功能需求

### 5.1 认证与用户

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| AUTH-01 | 用户可以请求注册或登录用途的邮箱验证码 | 同邮箱、同 IP、同用户维度限流；验证码只保存哈希；验证码有过期和失败次数限制 |
| AUTH-02 | 用户可以使用邮箱、用户名、密码和验证码注册 | 邮箱唯一；密码使用强密码哈希；成功后返回短期访问令牌 |
| AUTH-03 | 用户可以用密码或邮箱验证码登录 | 失败返回稳定错误码；失败次数受限；成功后更新最近登录时间 |
| AUTH-04 | 已登录用户可以读取和更新自己的资料 | Bearer Token 必须有效；禁用用户不可访问；响应不得包含密码哈希或内部敏感字段 |
| AUTH-05 | 已登录用户可以修改密码 | 必须校验当前密码；修改后可选择使旧 token 失效 |
| AUTH-06 | 系统支持生产会话策略 | 明确 token 有效期、刷新策略、退出登录策略和多端登录策略 |
| AUTH-07 | 系统支持账号风控 | 对异常登录、验证码爆破、Agent 高频调用和上传滥用进行限流或拦截 |

### 5.2 会话与消息

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| CHAT-01 | 用户可以创建、列出和读取自己的会话 | 只能访问本人资源；使用不可枚举 public_id；列表支持分页 |
| CHAT-02 | 用户可以向会话发送文本消息 | 只允许文本消息类型；长度限制明确；写入后刷新会话时间和缓存 |
| CHAT-03 | 消息可以带输入来源元数据 | 支持 `keyboard`、`voice` 等来源；未知来源被拒绝 |
| CHAT-04 | 消息可以绑定同一会话下已上传、未绑定的附件 | 附件归属、数量、状态和文件类型必须校验 |
| CHAT-05 | 用户可以读取会话消息列表 | 支持分页或游标；优先使用缓存；返回顺序稳定 |
| CHAT-06 | 用户可以管理历史会话 | 支持重命名、归档或删除会话；删除策略必须明确是否软删除 |
| CHAT-07 | 消息状态可感知 | 前端能区分发送中、生成中、完成、失败、可重试状态 |

### 5.3 Agent 问答

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| AGENT-01 | 用户通过 `/api/v1/agent/chat/stream` 发起流式问答 | 后端返回 SSE 事件：`user_message`、`agent_run`、`delta`、`done`、`error` |
| AGENT-02 | 每轮问答必须创建可追踪的 `agent_runs` 记录 | 保存输入快照、模型名、意图、工作流、输出快照、耗时、token 用量和失败原因 |
| AGENT-03 | Agent 回答前应装配上下文 | 包含近期消息、会话摘要、长期记忆、附件上下文、知识库 ID 和请求选项 |
| AGENT-04 | Agent 运行失败时应降级 | 未输出文本前可尝试下一个模型候选；全部失败后返回本地兜底回复 |
| AGENT-05 | 成功回答后应更新摘要和长期记忆 | 副作用失败不影响已完成回答；失败要回滚并记录日志 |
| AGENT-06 | Agent 应支持成本控制 | 记录 token、模型、工具调用次数；支持用户级、IP 级、全局级调用额度 |
| AGENT-07 | Agent 应防御提示注入 | 将用户输入、附件内容、RAG 结果和网页内容视为不可信上下文；工具调用必须受策略限制 |
| AGENT-08 | Agent 输出应可解释 | RAG 命中时展示引用来源；联网搜索结果与本地知识库结果应可区分 |

### 5.4 RAG 知识库

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| RAG-01 | 系统可以将本地 Markdown 菜谱数据索引到 Milvus | 索引脚本支持指定数据目录、知识库 ID、重建和增量更新 |
| RAG-02 | Agent 回答前根据问题决定是否检索 | 做菜、食材、菜谱类问题触发；寒暄和短控制指令跳过 |
| RAG-03 | 多轮问题可先改写再检索 | 查询改写失败时回退原问题，不阻断回答 |
| RAG-04 | 检索结果应保留引用信息 | 输出快照包含标题、分块序号、页码、分数和元数据 |
| RAG-05 | 知识库无命中时可补充联网搜索 | 只在配置 SerpApi 且 RAG miss 时追加 web search 上下文 |
| RAG-06 | RAG 应支持质量评估 | 使用 RAGAS 或人工样例集持续评估命中率、忠实度、答案相关性 |
| RAG-07 | RAG 应支持索引运维 | 支持 collection 版本、重建回滚、索引任务日志和失败重试 |

### 5.5 长期记忆

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| MEM-01 | 系统能识别用户偏好、忌口、厨具和健康目标 | 优先使用 LangChain structured output，失败时规则兜底 |
| MEM-02 | 记忆应去重或更新 | 同类型相似记忆不重复写入；显式更新时覆盖旧内容 |
| MEM-03 | 后续回答应注入相关记忆 | 上下文提供者从 MySQL 读取并按相关性筛选 |
| MEM-04 | 用户应可管理长期记忆 | 支持查看、编辑、删除或关闭记忆功能 |
| MEM-05 | 记忆应遵守隐私策略 | 不记录敏感身份信息；可配置保留期；用户删除后不可继续注入 |

### 5.6 附件与文档入库

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| FILE-01 | 用户可以在会话内上传一个或多个附件 | 限制大小、数量、扩展名、MIME、会话归属；文件名不可信 |
| FILE-02 | 用户可以删除未绑定附件 | 只能删除本人会话下未绑定消息的附件，并清理文件和解析记录 |
| FILE-03 | 文档附件可以被解析 | 解析任务有状态、超时、失败原因和重试入口 |
| FILE-04 | 解析后的文档可以进入 RAG 知识库 | 分块、向量化并写入 Milvus；返回入库数量和跳过原因 |
| FILE-05 | 文件处理必须安全 | 存储路径固定在安全目录；禁止路径穿越；限制危险类型；生产环境应接入恶意文件扫描 |
| FILE-06 | 长耗时解析应异步化 | 大文件解析、OCR、embedding 不阻塞聊天主请求 |

### 5.7 语音输入

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| VOICE-01 | 用户可以上传音频并获得转写文本 | 支持大小、时长、扩展名、MIME 校验 |
| VOICE-02 | 转写提供方可配置 | 支持 OpenAI 兼容语音接口或本地 faster-whisper |
| VOICE-03 | 语音不直接作为消息类型保存 | 正式消息仍为文本，输入来源通过 metadata 标识 |
| VOICE-04 | 语音转写体验可恢复 | 前端展示录音、上传、转写中、失败、重试状态 |
| VOICE-05 | 语音处理受限流保护 | 对单用户、单 IP 和全局并发做限制，避免音频任务拖垮服务 |

### 5.8 外部工具

| 编号 | 需求 | 生产验收要点 |
| --- | --- | --- |
| TOOL-01 | Agent 可调用 RAG、记忆、附件上下文、联网搜索和天气工具 | 工具由 `build_tools` 根据上下文和配置装配 |
| TOOL-02 | 外部 key 缺失时应温和降级 | 返回可理解提示，不抛出不可控异常 |
| TOOL-03 | 工具调用应可审计 | 记录工具名、耗时、失败原因和必要的输入摘要，不记录密钥 |
| TOOL-04 | 工具调用应受策略约束 | 禁止工具访问任意本地路径、任意 URL 或越权用户数据 |

## 6. 前端用户体验需求

| 编号 | 需求 | 验收要点 |
| --- | --- | --- |
| UX-01 | 首屏进入即是可用聊天工作台 | 登录后直接看到会话、消息区和输入区，不用经过营销页 |
| UX-02 | 流式回复体验顺滑 | 首 token 到达后立即渲染；支持中断、失败提示和重新生成 |
| UX-03 | 附件体验完整 | 展示上传进度、文件大小、解析状态、失败原因和删除入口 |
| UX-04 | 语音体验完整 | 展示录音状态、时长、转写进度、可编辑转写文本 |
| UX-05 | 历史会话易用 | 支持搜索、重命名、删除/归档、空状态和加载状态 |
| UX-06 | 错误信息可理解 | 用户看到行动建议；开发调试信息只进入日志，不直接暴露 |
| UX-07 | 移动端可用 | 聊天列表、输入区、附件栏、设置面板在移动端不遮挡、不溢出 |
| UX-08 | 可访问性基础达标 | 表单、按钮、状态提示支持键盘操作和可读标签 |

| UX-10 | 敏感操作有确认 | 删除会话、删除记忆、清空历史等操作需要确认和可理解反馈 |

## 7. 生产安全需求

### 7.1 应用与部署安全

| 编号 | 需求 | 验收要点 |
| --- | --- | --- |
| SEC-APP-01 | 生产环境必须关闭 debug 和 auto-reload | `DEBUG=false`；启动命令不使用 `--reload` |
| SEC-APP-02 | OpenAPI 文档在生产环境必须关闭或受保护 | `/docs`、`/redoc`、`/openapi.json` 不能无鉴权暴露 |
| SEC-APP-03 | 必须配置 Host header allowlist | 只允许生产域名和可信内部域名 |
| SEC-APP-04 | CORS 必须最小授权 | 明确前端域名 allowlist，不使用带凭证的 `*` |
| SEC-APP-05 | 安全响应头必须配置 | 至少包含 CSP、frame-ancestors 或 X-Frame-Options、nosniff、Referrer-Policy；可由网关统一设置 |
| SEC-APP-06 | 反向代理信任边界必须明确 | 只信任来自已知代理的 `X-Forwarded-*` 头 |

### 7.2 认证、授权与数据访问

| 编号 | 需求 | 验收要点 |
| --- | --- | --- |
| SEC-AUTH-01 | 所有业务接口默认需要认证 | 除健康检查、登录注册、验证码外，接口必须依赖当前用户 |
| SEC-AUTH-02 | 对象级授权必须逐接口校验 | conversation、message、attachment、memory、agent_run 都必须校验归属 |
| SEC-AUTH-03 | 公开资源 ID 不得使用自增 ID | 对外只暴露不可枚举 public_id |
| SEC-AUTH-04 | JWT 校验必须严格 | 校验签名、算法、过期时间；payload 不放密钥或敏感数据 |
| SEC-AUTH-05 | 密码和验证码必须使用安全哈希 | 不保存明文；响应模型不返回哈希字段 |

### 7.3 输入、上传与前端安全

| 编号 | 需求 | 验收要点 |
| --- | --- | --- |
| SEC-IN-01 | 请求体必须 schema 驱动 | 写接口使用明确 Pydantic schema；拒绝多余危险字段 |
| SEC-IN-02 | 上传必须限制资源消耗 | 限制单文件大小、总大小、文件数、请求体大小、解析超时 |
| SEC-IN-03 | 文件路径必须安全 | 用户文件名不能参与真实路径拼接；所有文件限制在上传根目录内 |
| SEC-IN-04 | Markdown 和富文本渲染必须防 XSS | 前端不得渲染未清洗 HTML；如启用 raw HTML 必须使用可靠 sanitizer |
| SEC-IN-05 | 前端不得嵌入密钥 | 浏览器环境变量只允许公开配置；第三方私钥只能在后端使用 |
| SEC-IN-06 | Token 存储策略必须明确 | 避免长期 token 持久化在 localStorage；如使用 cookie，必须设计 CSRF 防护 |

### 7.4 日志、隐私与供应链

| 编号 | 需求 | 验收要点 |
| --- | --- | --- |
| SEC-LOG-01 | 日志不得记录密钥和完整敏感内容 | API key、密码、验证码、JWT、SMTP 密码、数据库 URL 必须脱敏 |
| SEC-LOG-02 | Agent 快照要控制隐私暴露 | 可追踪但不过度保存用户隐私；支持按用户删除或脱敏 |
| SEC-DEP-01 | 依赖必须可审计 | 后端 pip、前端 npm 依赖锁定版本并纳入漏洞扫描 |
| SEC-DEP-02 | 镜像必须最小化 | 生产镜像不包含 `.env`、测试缓存、node_modules 开发依赖和本地密钥 |
| SEC-DEP-03 | 密钥必须由环境或密钥管理服务注入 | 不提交 `.env`；不同环境使用不同密钥；支持轮换 |

## 8. 效率与性能需求

| 编号 | 需求 | 目标或验收方式 |
| --- | --- | --- |
| PERF-01 | 普通 API 响应低延迟 | 登录、会话、消息列表在缓存命中时 p95 小于 300ms，缓存未命中时 p95 小于 800ms |
| PERF-02 | Agent 首 token 尽快返回 | 正常模型可用时首 token p95 小于 3s；RAG 检索不应明显阻塞流式启动 |
| PERF-03 | 长耗时任务异步化 | 附件解析、OCR、embedding、批量索引进入后台任务或独立 worker |
| PERF-04 | 数据库查询可扩展 | 高频查询字段必须有索引；列表接口分页；避免 N+1 查询 |
| PERF-05 | Redis 缓存可控 | 用户、会话、消息、RAG 查询使用 TTL；写入后主动失效 |
| PERF-06 | Milvus 检索性能可观测 | 记录向量召回耗时、rerank 耗时、命中数量和失败率 |
| PERF-07 | 外部 API 有超时与重试 | 模型、搜索、天气、语音接口必须有超时、有限重试和熔断策略 |
| PERF-08 | 成本可控 | 记录 token、模型、工具调用和语音/解析耗时；支持日/月预算告警 |
| PERF-09 | 前端资源体积受控 | 生产构建压缩，首屏资源按需加载，避免不必要的大依赖 |

## 12. 数据需求

系统当前核心数据对象：

- `users`：账号身份、邮箱、密码哈希、状态和登录时间。
- `email_verification_codes`：验证码哈希、用途、过期时间和失败次数。
- `conversations`：用户会话、标题、状态和最近消息时间。
- `conversation_summaries`：长会话滚动摘要及覆盖范围。
- `messages`：用户/助手/系统消息、文本内容、状态和扩展元数据。
- `attachments`：上传文件、会话/消息绑定、类型、存储路径、哈希和解析状态。
- `parse_results`：附件解析文本、结构化结果、OCR 结果和向量化状态。
- `agent_runs`：每轮 Agent 执行的输入输出快照、状态、模型、错误和耗时。
- `memory_items`：用户长期偏好、忌口、厨具、健康目标等记忆。
- Milvus collection：菜谱或用户文档分块向量、知识库 ID、文档元数据和分数。

生产需要补充的数据能力：

- 数据库迁移版本表和 migration 历史。
- 异步任务表或任务队列状态。
- 用户级配额、用量统计和成本统计。
- 审计日志或安全事件记录。
- 用户记忆管理状态，例如禁用、删除、更新时间。
- 文件生命周期字段，例如过期时间、删除时间、扫描状态。

## 14. 生产验收标准

生产上线前必须满足：

- 后端生产配置启动，`DEBUG=false`，无 auto-reload，`GET /health` 和 `GET /ready` 通过。
- 数据库 migration 可从空库创建完整 schema，也可从上一版本平滑升级。
- 用户可以完成验证码、注册、登录、创建会话、发送消息、流式 Agent 回复和读取历史闭环。
- 所有用户资源接口完成对象级授权测试，跨用户访问返回 403 或 404。
- Agent 模型不可用时返回本地兜底回复，并在 `agent_runs` 中记录失败细节。
- 做菜类问题触发 RAG 检索，输出快照和前端展示中包含引用来源。
- 用户可查看和删除长期记忆，删除后后续回答不再注入。
- 附件上传、解析、入库有状态可查，失败可重试，危险文件被拒绝。
- 前端在桌面和移动端均无布局遮挡，关键操作有加载、失败和重试反馈。
- 日志、指标和告警能定位一次请求、一次 Agent run 和一次外部 API 失败。
- MySQL、Redis、Milvus、上传文件和配置密钥有备份、恢复和轮换策略。
- CI/CD 至少执行后端测试、前端构建、依赖扫描和镜像启动检查。

## 15. 迭代路线建议

1. **生产基础设施阶段**：Docker、环境配置、Alembic、健康检查、CORS/Host/安全头、结构化日志。
2. **安全与权限阶段**：对象级授权补测、上传防护、生产限流、密钥管理、依赖扫描、日志脱敏。
3. **体验闭环阶段**：会话管理、记忆管理、附件解析状态、前端错误和重试、移动端适配。
4. **效率阶段**：后台任务队列、缓存策略完善、RAG 检索性能监控、token 成本统计。
5. **运维阶段**：指标、告警、备份恢复、发布回滚、staging 冒烟和压测。
6. **质量阶段**：RAGAS 固定评测集、端到端测试、提示词版本管理、模型路由优化。
