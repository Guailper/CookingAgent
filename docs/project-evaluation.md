# CookingAgent 项目评估报告

评估日期：2026-05-24  
评估对象：当前代码库、`docs/requirements-analysis.md`、`docs/architecture-design.md`  
评估目标：判断项目距离“可实际部署、面向生产环境运行”的差距，并给出优先级建议。

## 1. 结论摘要

CookingAgent 已经具备较完整的本地开发版智能烹饪助手基础：后端分层清晰，Agent、RAG、文件上传、语音转写、用户认证、对话存储、缓存和测试体系都有实现；前端也已具备聊天、上传、语音输入、流式响应等主要交互能力。

但从生产环境要求看，当前项目仍处于“功能型 MVP 向生产级系统过渡”的阶段。主要短板集中在部署运维、安全加固、异步任务、数据治理、可观测性、前端安全体验和生产级测试上。项目可以作为内部试用或受控环境演示使用，但不建议直接面向公网用户部署。

综合成熟度评估：

| 维度 | 评分 | 当前判断 |
| --- | ---: | --- |
| 核心业务功能 | 75/100 | 能完成智能问答、食谱生成、文件辅助、语音输入等核心路径 |
| Agent 与 RAG 能力 | 80/100 | 架构较完整，有长期记忆、检索、重排、快照、评测脚本基础 |
| 后端架构可维护性 | 78/100 | API、Service、Repository、Agent、RAG 分层较清楚 |
| 前端用户体验 | 65/100 | 核心聊天体验可用，但缺少任务状态、记忆管理、引用展示等生产级体验 |
| 安全性 | 45/100 | 有密码哈希和令牌机制，但存在本地存储密码、生产配置兜底、上传校验不足等问题 |
| 性能与扩展性 | 55/100 | 有缓存和部分限流，但缺少分页、异步任务、压测与资源隔离 |
| 部署与运维 | 25/100 | 缺少 Docker、CI/CD、迁移、就绪探针、指标和告警 |
| 数据治理与隐私 | 35/100 | 缺少导出、删除、保留策略、审计和敏感信息处理流程 |
| 测试质量 | 65/100 | 后端测试通过，前端可构建；缺少端到端、安全、性能和跨用户权限测试 |

建议当前整体定位为：**生产级架构雏形已形成，但生产上线前还需要一轮 P0 级工程补强**。

## 3. 与需求文档的符合度

### 3.1 已基本满足的需求

用户与认证：

- 已有注册、登录、刷新会话、发送邮箱验证码、重置密码等能力。
- 密码使用 PBKDF2-HMAC-SHA256 哈希，具备基础安全性。
- 后端多数业务接口已通过当前用户上下文做访问控制。

智能对话：

- 已实现会话、消息、Agent 流式响应和回答持久化。
- 支持 Agent 运行状态、失败回退、上下文快照、工具调用结果和用量提取。
- 支持基于长期记忆、RAG 和 Web 检索结果增强回答。

文件与语音：

- 文件上传已包含大小、数量、扩展名、归属校验、哈希记录和删除流程。
- 已有附件解析工作流和文档摄取流程。
- 前端具备附件上传、语音输入、流式消息展示和错误提示。

RAG：

- 已具备文本切分、嵌入、Milvus 存储、重排、缓存、检索器和 RAGAS 评测脚本。
- 架构上已为后续知识库扩展打下基础。

工程质量：

- 后端测试数量和覆盖面在 MVP 阶段较好。
- 前端可完成生产构建。
- 代码分层比普通原型项目更清晰，后续演进成本可控。

### 3.2 部分满足但需要增强的需求

生产安全：

- 认证基础可用，但前端将会话令牌存储在 `localStorage`，并且“记住我”逻辑会保存邮箱和密码，这不满足生产安全要求。
- 后端存在默认 `APP_SECRET_KEY=change-this-in-production` 的兜底配置，生产环境应强制拒绝启动。
- 限流依赖 Redis 可用性，Redis 关闭时退化为弱保护。
- IP 限流读取 `x-forwarded-for`，但没有可信代理边界校验。
- 缺少 CORS 白名单、Trusted Host、安全响应头、生产环境关闭调试文档等加固。

文件安全：

- 当前上传流程有扩展名和大小限制，但仍缺少 MIME 嗅探、恶意文件扫描、流式写入、对象存储隔离和解析沙箱。
- 文件读取采用一次性读入内存的方式，大文件或并发上传时存在内存压力。

异步任务：

- 架构文档要求生产级任务队列和可观测任务状态。
- 当前附件解析、文档摄取更接近同步或服务内流程，缺少独立 Worker、任务表、重试、进度、取消和死信处理。

用户体验：

- 聊天主流程可用，但缺少生产环境常见能力：会话重命名、删除、搜索、引用来源展示、记忆管理、任务进度、失败重试入口。
- 上传和解析结果之间缺少清晰的异步状态反馈。

数据治理：

- 当前有数据持久化基础，但缺少用户数据导出、彻底删除、保留周期、审计日志、隐私策略和敏感数据脱敏。

### 3.3 暂未满足的需求

部署与运维：

- 缺少 Dockerfile、docker-compose、生产环境配置模板、进程启动策略和反向代理说明。
- 缺少 CI/CD 流程。
- 缺少数据库迁移体系。当前存在 `AUTO_CREATE_TABLES`，适合开发环境，不适合生产演进。
- 缺少 readiness probe、metrics、tracing、集中日志、告警和备份恢复流程。

生产级质量保障：

- 缺少端到端测试、前端测试、安全测试、迁移测试、权限隔离测试、压测和故障演练。
- 缺少依赖漏洞扫描、密钥泄漏扫描、镜像扫描。

商业化和运营能力：

- 暂未看到用量配额、套餐限制、成本预算、用户行为审计、管理后台和运营指标。

## 4. 与架构设计文档的符合度

### 4.1 架构已落地的部分

后端分层：

- API 层、Service 层、Repository 层、Agent 层、RAG 层整体已经形成。
- 认证、会话、消息、附件、语音、Agent 调度等边界较清楚。

Agent 编排：

- 已有 Agent 运行器、工作流、上下文构建、工具快照、回答回退和状态记录。
- 对未来多工具、多模型和检索增强具备扩展空间。

RAG 子系统：

- Milvus、嵌入、重排、缓存、评测脚本等核心组件较完整。
- 从架构上看，RAG 已不是简单拼接，而是一个可演进子系统。

缓存：

- 已有 Redis 缓存管理和多类缓存键设计。
- 用户、会话、消息、RAG 等读路径具备优化基础。

### 4.2 架构文档中尚未落地的部分

部署架构：

- 文档描述了生产部署、容器化、反向代理、健康检查和运维能力，但代码库中尚未看到对应落地文件。

安全架构：

- 文档要求多层防护，但当前中间件层安全能力不足。
- 缺少全局安全策略、请求 ID、安全日志、审计日志和权限测试闭环。

任务架构：

- 文档中的异步任务、Worker、重试和任务状态设计尚未落地。
- 文件解析、知识入库、长任务处理仍需要任务化。

数据架构：

- 缺少迁移体系和生产数据生命周期管理。
- RAG 集合版本、索引升级、回滚和重建流程尚不完整。

可观测架构：

- 日志已有基础，但缺少指标、链路追踪、慢请求、错误聚合和告警。
- 无法支撑生产环境问题定位和容量规划。

## 5. 关键风险清单

### P0：上线阻断风险

1. 前端保存明文密码

`frontend/src/services/authService.ts` 的记住登录逻辑会将邮箱和密码保存到 `localStorage`。这是生产环境不可接受的安全风险。应立即移除密码保存，只允许保存邮箱、偏好或使用安全会话机制。

2. 会话令牌存储在 `localStorage`

`localStorage` 容易受到 XSS 影响。生产环境建议改为 HttpOnly、Secure、SameSite Cookie，或至少建立明确的 XSS 防护、刷新令牌轮换和失效策略。

3. 生产密钥存在默认兜底值

`APP_SECRET_KEY` 有开发默认值。生产环境必须强制通过环境变量提供，并在默认值、弱密钥或过短密钥时拒绝启动。

4. 缺少生产部署闭环

当前缺少 Docker、迁移、CI/CD、启动脚本、健康探针和回滚策略。即使功能可用，也难以稳定上线。

5. 缺少数据库迁移体系

生产系统不能依赖自动建表。应恢复并完善 Alembic 或同等迁移方案，提供基线迁移、升级、回滚和迁移测试。

### P1：高优先级风险

1. 文件上传安全不足

扩展名校验不足以防御伪装文件。应增加 MIME 嗅探、内容扫描、文件隔离、解析沙箱和对象存储策略。

2. 限流策略不够稳固

Redis 不可用时限流退化，且 IP 获取信任 `x-forwarded-for`。生产环境应明确反向代理链路，并加入用户级、IP 级、接口级、模型调用级限流。

3. 长任务没有任务状态

附件解析、文档入库和 RAG 重建应异步化，并提供任务进度、重试、失败原因和取消能力。

4. 缺少跨用户权限测试

虽然代码中已有对象归属校验，但需要通过测试覆盖“用户 A 访问用户 B 会话、消息、附件”的所有路径。

5. 缺少分页和容量控制

会话、消息、记忆等列表接口如果一次性返回全部数据，会在真实用户量增长后影响性能和体验。

### P2：中期演进风险

1. 缺少成本治理

Agent、RAG、Web 检索和语音转写会带来外部服务成本。需要用量记录、预算、配额、熔断和降级。

2. 缺少数据治理

需要补充用户数据导出、删除、保留周期、脱敏、审计和隐私合规说明。

3. 缺少 RAG 版本治理

知识库索引升级、重建、回滚、评测门禁和灰度发布需要进一步产品化。

4. 前端缺少生产级细节体验

包括引用来源、解析状态、记忆开关、失败重试、会话管理、移动端细节和可访问性。

## 6. 建议整改路线

### 阶段一：生产上线前必须完成

目标：让项目从“能跑”变成“可部署、可回滚、可防护”。

- 移除前端密码本地保存逻辑。
- 重新设计会话存储，优先使用 HttpOnly Cookie。
- 增加生产配置校验，弱密钥、默认密钥、缺失关键配置时拒绝启动。
- 增加 CORS 白名单、Trusted Host、安全响应头和生产环境关闭调试文档。
- 增加 `/ready`，检查数据库、Redis、Milvus、模型配置等关键依赖。
- 增加 Dockerfile、docker-compose 或生产部署模板。
- 建立 Alembic 迁移基线，生产禁用自动建表。
- 增加 CI：后端测试、前端构建、Lint、类型检查、依赖扫描、密钥扫描。
- 增加跨用户权限测试和核心接口集成测试。

### 阶段二：生产体验和稳定性补强

目标：让系统可以承受真实用户和真实文件。

- 引入任务队列和 Worker，任务覆盖附件解析、文档入库、RAG 重建。
- 增加任务表、任务状态 API、重试、失败原因和进度通知。
- 文件上传改为流式写入，增加 MIME 嗅探、恶意文件扫描和对象存储适配。
- 会话、消息、记忆、任务列表增加分页、排序和游标。
- 增加 Prometheus 指标、结构化日志、请求 ID、慢请求日志和错误告警。
- 增加 Agent 用量、模型成本、用户配额和熔断降级。

### 阶段三：产品化和长期治理

目标：让项目具备长期运营能力。

- 增加记忆管理界面：查看、编辑、删除、禁用、导出。
- 增加引用来源展示、食谱结构化卡片、营养/过敏原提示和用户偏好解释。
- 增加用户数据导出、注销、删除、保留策略和审计日志。
- 建立 RAG 版本、评测门禁、索引回滚和知识库管理后台。
- 增加端到端测试、安全测试、性能测试和故障演练。
- 增加管理后台、运营指标和用户反馈闭环。

## 7. 推荐验收标准

生产上线前建议至少满足以下标准：

- 后端测试、前端构建、Lint、类型检查全部在 CI 中通过。
- 所有业务接口通过鉴权和对象归属测试。
- 默认密钥、弱密钥、调试模式在生产环境无法启动。
- 前端不保存明文密码。
- 生产接口具备 CORS、Host、安全响应头和限流。
- 数据库结构由迁移工具管理。
- 核心依赖可通过 `/ready` 检查。
- 文件上传具备大小、类型、内容、扫描和隔离策略。
- Agent 调用具备用量记录、超时、失败回退和成本上限。
- 关键错误可在日志、指标和告警中定位。
- 至少完成一轮真实部署演练和恢复演练。

## 8. 总体建议

项目的 AI 能力和后端业务基础已经比较扎实，值得继续投入。当前最不应该优先扩展的是更多炫目的 Agent 功能；更应该先补齐生产工程底座。

建议下一步按照以下顺序推进：

1. 先修复前端本地保存密码和生产密钥校验问题。
2. 再补齐 Docker、迁移、CI、健康检查和安全中间件。
3. 然后将文件解析和 RAG 入库任务异步化。
4. 最后增强数据治理、可观测性和前端生产体验。

完成阶段一后，项目可以进入受控生产试运行；完成阶段二后，才更适合开放给真实用户持续使用。
