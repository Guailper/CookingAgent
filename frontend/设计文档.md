# 前端设计文档

## 1. 文档目的

这份文档描述当前 `frontend/` 目录下已经落地的前端结构、模块职责，以及前后端已经接通到什么程度。

当前前端已经实现两类页面：

- 认证页：登录、注册、忘记密码入口
- 主界面：登录后进入的聊天工作台

当前前端已经与后端完成以下真实接入：

- 登录
- 注册
- 当前用户信息恢复
- 会话列表加载
- 单个会话详情加载
- 会话消息列表加载
- 创建新会话
- 发送用户消息

当前仍然保留在前端增强层的部分包括：

- 新对话推荐气泡
- 顶部搜索结果整理
- 设置菜单与设置面板内容
- AI 助手回复生成

## 3. 分层职责

### `App.tsx`

应用最外层入口。

这里只负责承接 `router/`，不再放页面状态、接口调用和组件细节。

### `main.tsx`

浏览器挂载入口。

这里只负责挂载 React 根节点，并加载全局样式 `styles/global.css`。

### `router/`

负责页面切换和登录态恢复。

当前没有接入 `react-router-dom`，所以先用轻量本地路由控制认证页和主界面之间的切换：

- 未登录时进入 `LoginPage`
- 本地存在 session 时，先调用后端 `/auth/me` 校验 token 是否仍然有效
- 校验成功后进入 `ChatPage`
- 校验失败后清空本地 session 并回到 `LoginPage`

### `pages/`

负责页面级组装。

页面只做“把组件和 hook 组合起来”的工作，不直接承担复杂的数据编排逻辑。

当前页面包括：

- `LoginPage.tsx`：认证页容器
- `ChatPage.tsx`：登录后主界面容器

### `types/`

负责纯类型定义。

当前分为两层：

- `types/api.ts`
  - 定义后端接口的原始返回结构
  - 方便联调时对照字段名
- `types/auth/` 和 `types/chat/`
  - 定义服务层加工后的前端类型
  - 供 hooks 和组件直接使用

### `services/`

负责服务层逻辑。

当前分为：

- `services/auth/authService.ts`
  - 登录
  - 注册
  - 当前用户信息获取
  - 本地 session / remember 信息读写
- `services/chat/chatService.ts`
  - 调用后端会话接口
  - 调用后端消息接口
  - 把后端原始结构转换成前端友好结构
  - 提供设置菜单、推荐气泡和搜索结果整理这类前端增强逻辑

### `hooks/`

负责页面交互状态和业务编排。

当前分为：

- `hooks/auth/useAuth.ts`
  - 管理登录和注册表单状态
  - 处理认证成功后的跳转回调
- `hooks/chat/useWorkspace.ts`
  - 管理侧栏展开/收起
  - 管理设置菜单和设置面板
  - 加载会话列表
  - 打开会话并加载消息
  - 创建会话并发送消息
  - 管理顶部搜索和推荐气泡点击行为

### `components/`

负责 UI 组件拆分。

当前分为：

- `components/auth/`：认证页 UI
- `components/chat/`：主界面 UI

组件原则：

- 组件尽量只负责展示和事件透传
- 状态尽量上收至 hook
- 每个核心文件顶部写清楚职责说明

### `styles/`

负责样式分层。

当前样式文件为：

- `global.css`：全局基础样式与通用状态提示样式
- `auth.css`：认证页样式
- `chat.css`：主界面样式

- 侧栏整体以当前视口高度为上限，顶部品牌区和底部用户区固定在栏内，不参与会话列表滚动

### 5.4 聊天消息区

对应组件：`components/chat/MessageList.tsx`

- 当前会话摘要
- 最后更新时间
- 消息气泡列表

说明：

当前后端只提供用户消息写入与读取，因此这部分还没有接入真实 AI 助手回复。

当前消息区样式补充约束：

- 单条消息气泡按内容宽度收缩显示，只保留最大宽度上限，不再默认占据大半个消息区

- 消息正文保留自然换行，超长文本需要自动换行，避免撑破气泡布局
- 主工作区采用“顶部工具栏 / 可选提示条 / 消息滚动区 / 底部输入区”的分层结构，只有消息滚动区允许纵向滚动

### 5.5 输入框

对应组件：`components/chat/MessageComposer.tsx`

功能包括：

- 输入新消息
- 回车发送
- 点击发送按钮发送
- 在发送中禁用按钮，避免重复提交

行为规则：

- 当前已打开会话时，消息会写入该会话
- 当前处于欢迎态时，前端会先请求后端创建会话，再发送第一条消息
- 输入框容器固定在主工作区底部，消息数量增加时只能推动消息区内部滚动，不能把输入框挤出当前视口

## 6. 主界面状态流

主界面主要状态集中在 `hooks/chat/useWorkspace.ts` 中。

当前核心状态包括：

- `conversations`
  - 保存后端返回的会话列表
- `activeConversationId`
  - 当前打开的会话 id
- `draft`
  - 输入框草稿内容
- `searchKeyword`
  - 顶部搜索框当前输入
- `workspaceNotice`
  - 主界面的请求失败提示
- `isSendingMessage`
  - 控制发送消息时的重复提交

主界面的核心流程如下：

1. 用户登录成功
2. `router/index.tsx` 切换到 `ChatPage`
3. `useWorkspace.ts` 调用后端加载会话列表
4. 用户点击某个会话时，再加载该会话的消息列表
5. 用户在欢迎态发送第一条消息时：
   - 先创建会话
   - 再发送消息
   - 再把该会话移动到侧栏顶部
6. 用户在已有会话中发送消息时：
   - 直接调用消息接口
   - 更新本地会话列表和消息区展示
7. 如果后端返回鉴权失败：
   - `useWorkspace.ts` 触发退出登录
   - 路由回到认证页

## 7. 注释规范

为了让不是当前开发者的人也能快速参与开发，当前代码采用以下注释约定：

- 每个核心文件顶部必须写“这个文件负责什么”
- 复杂组件内部要在关键区域前写一句注释，说明这一段 UI 的目的
- hook 中的关键状态要写注释，说明它控制哪部分交互
- 服务层要说明哪些逻辑是后端真实请求，哪些仍然是前端增强层

不建议写的注释：

- “给变量赋值”
- “渲染一个 div”
- 只重复语法、不解释业务意图的注释

建议优先解释：

- 为什么这样拆
- 这一段状态或函数负责哪条业务链路
- 后续应当从哪里继续扩展

## 11. 语音输入与文件上传扩展设计

### 11.1 本轮范围

- 语音功能先定义为“语音输入转文字”：用户在 `MessageComposer` 中录音，前端把音频提交给后端转写接口，拿到转写结果后再回填输入框。
- 本轮不包含 AI 语音播报、音频消息播放器、实时双工语音对话。
- 文件上传先定义为“消息附件”：用户在发送一条消息时可附带文档或图片，后端返回 `attachmentIds` 后，再把它们绑定到当前消息。
- 设计目标是尽量不推翻当前纯文本聊天链路，只在现有 “创建会话 -> 发送消息” 流程上做增强。

### 11.2 前端交互流

#### 11.2.1 语音输入流

1. 用户点击麦克风按钮开始录音。
2. 用户再次点击麦克风，或达到前端设定的录音上限后自动结束录音。
3. 前端拿到录音 `Blob` 后，调用 `POST /api/v1/voice/transcriptions`。
4. 后端返回 `transcript` 后，前端把内容写入 `draft`。
5. 用户可以继续编辑转写文本，再沿用现有发送按钮提交。
6. 发送时在 `extra_metadata` 中带上 `input_source=voice`，用于后端区分消息来源。

#### 11.2.2 文件上传流

1. 用户通过附件按钮选择本地文件。
2. 前端先做扩展名、大小、数量的本地校验，不合法文件在发送前直接提示。
3. 合法文件先保存在前端 `pendingAttachments` 列表中，允许删除，不立即上传。
4. 用户点击发送时，如果当前还没有会话，前端先调用现有创建会话接口。
5. 前端调用 `POST /api/v1/conversations/{conversation_id}/attachments` 上传本地待发送文件。
6. 上传成功后，前端再调用 `POST /api/v1/conversations/{conversation_id}/messages`，并在请求体中传入 `attachment_ids`。
7. 附件上传和消息发送都成功后，再统一清空输入框与本地附件列表；任一步失败都保留草稿与本地附件，允许用户重试。

### 11.5 前端依赖的接口契约

#### 11.5.1 语音转写接口

```text
POST /api/v1/voice/transcriptions
Content-Type: multipart/form-data

fields:
- file: audio/webm | audio/wav | audio/mp3 | audio/m4a
- language: optional, default zh

response:
- transcript
- duration_ms
- mime_type
- file_size
```

#### 11.5.2 附件上传接口

```text
POST /api/v1/conversations/{conversation_id}/attachments
Content-Type: multipart/form-data

fields:
- files[]

response:
- attachments[]
  - public_id
  - original_name
  - file_size
  - mime_type
  - parse_status
```

#### 11.5.3 发送消息接口

```json
{
  "content": "帮我总结这份文档",
  "message_type": "text",
  "attachment_ids": ["att_xxx", "att_yyy"],
  "extra_metadata": {
    "input_source": "voice"
  }
}
```
