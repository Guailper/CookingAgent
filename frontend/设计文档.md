# 前端设计文档

## 1. 文档目的

这份文档描述当前 `frontend/` 目录下已经落地的前端结构、模块职责，以及前后端已经接通到什么程度。

当前前端已经实现两类页面：

- 认证页：登录、注册、忘记密码入口
- 主界面：登录后进入的聊天工作台

当前前端已经与后端完成以下真实接入：

- 登录
- 注册
- 当前用户信息恢复
- 会话列表加载
- 单个会话详情加载
- 会话消息列表加载
- 创建新会话
- 发送用户消息

当前仍然保留在前端增强层的部分包括：

- 新对话推荐气泡
- 顶部搜索结果整理
- 设置菜单与设置面板内容
- AI 助手回复生成

## 3. 分层职责

### `App.tsx`

应用最外层入口。

这里只负责承接 `router/`，不再放页面状态、接口调用和组件细节。

### `main.tsx`

浏览器挂载入口。

这里只负责挂载 React 根节点，并加载全局样式 `styles/global.css`。

### `router/`

负责页面切换和登录态恢复。

当前没有接入 `react-router-dom`，所以先用轻量本地路由控制认证页和主界面之间的切换：

- 未登录时进入 `LoginPage`
- 本地存在 session 时，先调用后端 `/auth/me` 校验 token 是否仍然有效
- 校验成功后进入 `ChatPage`
- 校验失败后清空本地 session 并回到 `LoginPage`

### `pages/`

负责页面级组装。

页面只做“把组件和 hook 组合起来”的工作，不直接承担复杂的数据编排逻辑。

当前页面包括：

- `LoginPage.tsx`：认证页容器
- `ChatPage.tsx`：登录后主界面容器

### `types/`

负责纯类型定义。

当前分为两层：

- `types/api.ts`
  - 定义后端接口的原始返回结构
  - 方便联调时对照字段名
- `types/auth/` 和 `types/chat/`
  - 定义服务层加工后的前端类型
  - 供 hooks 和组件直接使用

### `services/`

负责服务层逻辑。

当前分为：

- `services/auth/authService.ts`
  - 登录
  - 注册
  - 当前用户信息获取
  - 本地 session / remember 信息读写
- `services/chat/chatService.ts`
  - 调用后端会话接口
  - 调用后端消息接口
  - 把后端原始结构转换成前端友好结构
  - 提供设置菜单、推荐气泡和搜索结果整理这类前端增强逻辑

### `hooks/`

负责页面交互状态和业务编排。

当前分为：

- `hooks/auth/useAuth.ts`
  - 管理登录和注册表单状态
  - 处理认证成功后的跳转回调
- `hooks/chat/useWorkspace.ts`
  - 管理侧栏展开/收起
  - 管理设置菜单和设置面板
  - 加载会话列表
  - 打开会话并加载消息
  - 创建会话并发送消息
  - 管理顶部搜索和推荐气泡点击行为

### `components/`

负责 UI 组件拆分。

当前分为：

- `components/auth/`：认证页 UI
- `components/chat/`：主界面 UI

组件原则：

- 组件尽量只负责展示和事件透传
- 状态尽量上收至 hook
- 每个核心文件顶部写清楚职责说明

### `styles/`

负责样式分层。

当前样式文件为：

- `global.css`：全局基础样式与通用状态提示样式
- `auth.css`：认证页样式
- `chat.css`：主界面样式

## 5. 主界面当前实现说明

主界面位于 `pages/ChatPage.tsx`，由以下模块组成。

### 5.1 左侧菜单栏

对应组件：`components/chat/ChatSidebar.tsx`

功能包括：

- 展示品牌信息
- 展示“开启新对话”按钮
- 展示最近会话列表
- 展示用户头像与邮箱
- 展示设置按钮
- 在设置按钮上方弹出设置菜单

说明：

当前最近会话列表来自后端真实接口，不再是本地 mock 数据。

当前列表样式补充约束：

- 每条会话记录只展示一行标题
- 标题过长时使用省略号截断，不允许撑高单条记录
- 会话列表在侧栏可用高度内独立滚动，不再用固定很小的最大高度裁切
- 侧栏整体以当前视口高度为上限，顶部品牌区和底部用户区固定在栏内，不参与会话列表滚动

### 5.2 顶部工具栏

对应组件：`components/chat/WorkspaceTopBar.tsx`

功能包括：

- 系统标题和说明
- 搜索输入框
- 半透明搜索结果列表
- 通知与帮助按钮

说明：

当前搜索结果是前端基于“已加载会话 + 推荐气泡”做的本地整理，不是独立后端搜索接口。

### 5.3 欢迎态与推荐气泡

对应组件：`components/chat/PromptSuggestions.tsx`

在“新对话”状态下，主区域会展示：

- 欢迎标题
- 引导文案
- 推荐对话气泡

说明：

推荐气泡仍然保留在前端层，点击后会触发真实后端创建会话和发送消息。

### 5.4 聊天消息区

对应组件：`components/chat/MessageList.tsx`

进入某段会话后，右侧主区域会展示：

- 当前会话标题
- 当前会话摘要
- 最后更新时间
- 消息气泡列表

说明：

消息列表已经来自后端真实接口。

当前后端只提供用户消息写入与读取，因此这部分还没有接入真实 AI 助手回复。

当前消息区样式补充约束：

- 单条消息气泡按内容宽度收缩显示，只保留最大宽度上限，不再默认占据大半个消息区
- 消息头部左侧显示发送者，右上角显示该条消息时间
- 消息正文保留自然换行，超长文本需要自动换行，避免撑破气泡布局
- 主工作区采用“顶部工具栏 / 可选提示条 / 消息滚动区 / 底部输入区”的分层结构，只有消息滚动区允许纵向滚动

### 5.5 输入框

对应组件：`components/chat/MessageComposer.tsx`

功能包括：

- 输入新消息
- 回车发送
- 点击发送按钮发送
- 在发送中禁用按钮，避免重复提交

行为规则：

- 当前已打开会话时，消息会写入该会话
- 当前处于欢迎态时，前端会先请求后端创建会话，再发送第一条消息
- 输入框容器固定在主工作区底部，消息数量增加时只能推动消息区内部滚动，不能把输入框挤出当前视口

### 5.6 设置菜单与设置面板

对应组件：

- `components/chat/ChatSidebar.tsx`
- `components/chat/SettingsPanel.tsx`

当前设置菜单提供三类入口：

- 账号设置
- 界面偏好
- 通知设置

说明：

当前设置内容仍然是前端静态占位，用于承接 UI 结构，后续再接真实设置接口。

## 6. 主界面状态流

主界面主要状态集中在 `hooks/chat/useWorkspace.ts` 中。

当前核心状态包括：

- `conversations`
  - 保存后端返回的会话列表
- `activeConversationId`
  - 当前打开的会话 id
- `draft`
  - 输入框草稿内容
- `searchKeyword`
  - 顶部搜索框当前输入
- `workspaceNotice`
  - 主界面的请求失败提示
- `isSendingMessage`
  - 控制发送消息时的重复提交

主界面的核心流程如下：

1. 用户登录成功
2. `router/index.tsx` 切换到 `ChatPage`
3. `useWorkspace.ts` 调用后端加载会话列表
4. 用户点击某个会话时，再加载该会话的消息列表
5. 用户在欢迎态发送第一条消息时：
   - 先创建会话
   - 再发送消息
   - 再把该会话移动到侧栏顶部
6. 用户在已有会话中发送消息时：
   - 直接调用消息接口
   - 更新本地会话列表和消息区展示
7. 如果后端返回鉴权失败：
   - `useWorkspace.ts` 触发退出登录
   - 路由回到认证页

## 7. 注释规范

为了让不是当前开发者的人也能快速参与开发，当前代码采用以下注释约定：

- 每个核心文件顶部必须写“这个文件负责什么”
- 复杂组件内部要在关键区域前写一句注释，说明这一段 UI 的目的
- hook 中的关键状态要写注释，说明它控制哪部分交互
- 服务层要说明哪些逻辑是后端真实请求，哪些仍然是前端增强层

不建议写的注释：

- “给变量赋值”
- “渲染一个 div”
- 只重复语法、不解释业务意图的注释

建议优先解释：

- 为什么这样拆
- 这一段状态或函数负责哪条业务链路
- 后续应当从哪里继续扩展

## 9. 后续扩展建议

### 9.1 接入真实 AI 回复

建议优先从以下位置扩展：

- 后端新增助手消息生成接口
- `services/chat/chatService.ts` 增加助手回复请求
- `hooks/chat/useWorkspace.ts` 在发送用户消息后继续请求助手消息

尽量保持这些组件不变：

- `ChatPage.tsx`
- `ChatSidebar.tsx`
- `MessageList.tsx`
- `MessageComposer.tsx`

### 9.2 接入真实设置接口

建议按下面顺序扩展：

1. 后端增加设置读取与保存接口
2. `services/chat/chatService.ts` 或新增 `services/settings/`
3. `hooks/chat/useWorkspace.ts` 或新增 `hooks/settings/`
4. `SettingsPanel.tsx` 只保留展示和表单输入

### 9.3 引入正式路由

当前页面不多，所以先用轻量路由控制。

如果后续出现这些情况，就建议升级到 `react-router-dom`：

- 页面数量明显增加
- 需要地址栏直达某个页面
- 需要权限路由和嵌套路由

## 10. 当前阶段说明

当前前端已经从“认证真实接口 + 主界面 mock 数据”演进到“认证真实接口 + 主界面核心会话链路真实接口”。

因此，后续开发建议优先遵循：

- 先补后端接口能力
- 再补服务层映射
- 最后补 hooks 和 UI 细节
